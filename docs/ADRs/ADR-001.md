# ADR-001: Hybrid Search Architecture (FTS5 + HNSW + RRF)

- **Status**: Proposed
- **Date**: 2026-03-23

## Context

The Emailibrium search system must handle both semantic similarity queries ("emails about quarterly budget planning") and exact-match queries ("invoice #INV-2024-0847"). Pure dense retrieval (vector search alone) is known to underperform BM25 on out-of-domain and exact-match tasks. Thakur et al. (2021) demonstrated that zero-shot dense retrieval often fails to outperform BM25 on out-of-domain benchmarks (BEIR), particularly for keyword-heavy or entity-centric queries common in email search. A hybrid approach combining both paradigms addresses this fundamental limitation.

The plan proposes combining SQLite FTS5 full-text search with HNSW vector search, fusing results via Reciprocal Rank Fusion (RRF). SONA adaptive re-ranking is applied as a final personalization stage.

## Decision

Implement hybrid search combining FTS5 keyword search and HNSW vector search, fused via Reciprocal Rank Fusion (RRF) with k=60. SONA re-ranking is applied as the final stage for personalization.

### Search Pipeline

1. **Step 1**: Generate query embedding (~5ms)
2. **Step 2**: HNSW vector search (~2ms) -- runs concurrently with Step 3
3. **Step 3**: FTS5 full-text search (~10ms) -- runs concurrently with Step 2
4. **Step 4**: RRF fusion of results from Steps 2 and 3 (~2ms)
5. **Step 5**: SONA adaptive re-ranking (~1ms)

Steps 2 and 3 are parallelized via `tokio::join!` to minimize total latency.

### RRF Formula

```
score(d) = SUM( 1 / (k + rank_i(d)) )  where k = 60
```

For each document `d`, sum the reciprocal of its rank plus `k` across each retrieval method `i`. The constant k=60 follows Cormack et al. (2009) and prevents top-ranked documents from dominating excessively.

### Configuration

- Default weights: 60% semantic / 40% keyword (adjustable per-user via SONA learning)
- Target total latency: ~20ms (5ms embed + 2ms HNSW + 10ms FTS5 + 3ms fusion + SONA)
- Top-K candidates from each method: 100 (configurable)
- Final result set: 20 documents returned to the UI

### Validation Requirements

Ablation testing is required before launch to quantify each component's contribution:

1. FTS5 alone (baseline keyword search)
2. HNSW vector search alone (baseline semantic search)
3. Hybrid (FTS5 + HNSW + RRF) without SONA
4. Full hybrid (FTS5 + HNSW + RRF + SONA)

Metrics: Precision@10, Recall@10, NDCG@10, MRR, and p95 latency for each configuration.

## Consequences

### Positive

- Addresses the known weakness of dense retrieval on exact-match and entity-centric queries
- RRF is training-free and parameter-light (only k to tune), reducing operational complexity
- Parallelized execution keeps latency competitive with single-method search
- Configurable weights allow per-user tuning via SONA feedback loop
- Ablation testing provides empirical evidence for each component's value

### Negative

- Two indices (FTS5 + HNSW) must be maintained, doubling index maintenance cost on writes
- FTS5 index in SQLite and HNSW index in RuVector must stay synchronized -- write failures to one index without the other create inconsistency
- More complex query pipeline increases debugging difficulty
- RRF assumes rank-level fusion is sufficient; score-level fusion (e.g., CombSUM) might outperform in some scenarios but requires score normalization

### Neutral

- Total storage increases modestly: FTS5 index is typically 30-50% of raw text size, HNSW index is N _ 384 _ 4 bytes for float32 vectors
- The k=60 constant may benefit from tuning on real email data but is a well-established default

## Alternatives Considered

### Pure Vector Search (HNSW only)

- **Pros**: Simpler architecture, single index, lower maintenance cost, captures semantic meaning
- **Cons**: Poor performance on exact-match queries (invoice numbers, email addresses, specific dates), confirmed by Thakur et al. 2021 BEIR benchmarks

### BM25 / FTS5 Only

- **Pros**: Simplest implementation, well-understood ranking, excellent for keyword queries
- **Cons**: No semantic understanding -- "budget discussion" would not match "financial planning conversation", vocabulary mismatch problem unaddressed

### Learned Fusion (trained model to combine scores)

- **Pros**: Potentially optimal fusion weights learned from data, can capture non-linear interactions
- **Cons**: Requires labeled training data we do not have at launch, adds model inference latency, risk of overfitting to training distribution

### Cross-Encoder Re-ranking

- **Pros**: State-of-the-art re-ranking quality, captures fine-grained query-document interaction
- **Cons**: Too slow for local execution (~100ms per document pair), incompatible with 20ms latency target without GPU
