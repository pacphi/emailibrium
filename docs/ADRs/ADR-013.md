# ADR-013: AI Model Lifecycle Management

- **Status**: Proposed
- **Date**: 2026-03-23
- **Extends**: ADR-011 (ONNX Runtime as Default Embedding Provider)
- **Research References**: docs/research/llm-options.md -- Sections 3.4, 5.2, 8.1, 9.3, 10

## Context

ADR-011 establishes ONNX-based embedding via `fastembed` as the default provider, with models auto-downloaded from Hugging Face Hub on first run. This introduces a model lifecycle that must be managed: download, integrity verification, caching, disk space, model switching, and re-embedding when models change.

Without explicit lifecycle management:

1. **First-run experience is opaque:** Users see a delay on first launch with no explanation of what is being downloaded or why.
2. **Integrity is unverified:** A corrupted or tampered model file could produce incorrect embeddings silently.
3. **Model switching is destructive:** If a user changes the configured model, existing email vectors become incompatible with the new model's embedding space, breaking search quality without any indication.
4. **Disk space accumulates:** Multiple model downloads over time consume disk space with no cleanup mechanism.
5. **Offline users are blocked:** Users who never connect to the internet after installation cannot use the embedding system.

## Decision

Implement a model manager component that handles the full ONNX model lifecycle: download, verification, caching, model switching, re-embedding, and cleanup.

### Model Storage Layout

```
~/.emailibrium/models/
  onnx/
    all-MiniLM-L6-v2/
      model.onnx
      tokenizer.json
      config.json
      checksums.sha256
    bge-small-en-v1.5/
      model.onnx
      tokenizer.json
      config.json
      checksums.sha256
  clip/
    clip-vit-b-32-text/
      ...
    clip-vit-b-32-vision/
      ...
```

The base path (`~/.emailibrium/models/`) is configurable via `ai.embedding.onnx.model_path`. Models are organized by provider type and model name. Multiple models can coexist on disk (for switching or evaluation purposes).

### Download and Initialization

1. On application startup (if `download_on_startup: true`) or on first `embed()` call (if `download_on_startup: false`), check whether the configured model exists in the local cache.
2. If the model is not cached, download from Hugging Face Hub. Display a progress bar if `show_download_progress: true`. Log the download URL, expected size, and expected SHA-256 checksum.
3. After download completes, verify the SHA-256 checksum of every downloaded file against the manifest embedded in the Emailibrium binary. If verification fails, delete the downloaded files and return an error.
4. Initialize the `fastembed::TextEmbedding` instance with the verified model files.

### Integrity Verification

- SHA-256 checksums for all supported models are embedded in the Emailibrium binary at compile time (a static manifest).
- Every model file is verified against this manifest on download and on each application startup (before loading into ONNX Runtime).
- If a cached model file fails verification (corruption, tampering), it is deleted and re-downloaded on the next startup.
- ONNX models are declarative computation graphs with no code execution capability, which limits the attack surface compared to pickle-based model formats. Nonetheless, checksum verification is defense-in-depth.

### Model Switching and Re-embedding

When a user changes `ai.embedding.onnx.model` in the configuration (for example, from `all-MiniLM-L6-v2` to `bge-small-en-v1.5`), the following sequence occurs on the next application restart:

1. **Detect model change**: Compare the configured model name against the model name stored in the database metadata table. If they differ, a model switch has occurred.
2. **Download new model**: If the new model is not already cached, download and verify it.
3. **Load new model**: Initialize `fastembed::TextEmbedding` with the new model.
4. **Mark all embeddings as stale**: Update all email records to `embedding_status: 'stale'` in the database.
5. **Re-embed in background**: Process stale emails through the ingestion pipeline in background batches, generating new vectors with the new model.
6. **Maintain searchability during re-embedding**: Old vectors remain in the HNSW index and are searchable until replaced. Search quality degrades gradually (mixed old/new vectors) rather than failing entirely.
7. **Expose progress**: The re-embedding progress is available via `GET /api/v1/vectors/reindex-status`, returning the count of total, completed, and remaining emails.

**Important constraint:** Vectors from different models are not comparable. A vector generated by `all-MiniLM-L6-v2` has no meaningful cosine similarity with a vector generated by `bge-small-en-v1.5`, even if both are 384-dimensional. The re-embedding step is therefore mandatory when switching models.

**Dimension change handling:** If the new model has different dimensions (for example, switching from a 384D model to a 768D model), the HNSW index must be rebuilt entirely. The model manager detects dimension mismatches and triggers a full index rebuild rather than incremental re-embedding.

### Offline Mode

- If the configured model exists in the local cache, Emailibrium operates fully offline with no network calls.
- If the model is not cached and the network is unavailable, the model manager returns a clear error: "Embedding model 'all-MiniLM-L6-v2' not found in ~/.emailibrium/models/. Connect to the internet and restart, or run `emailibrium --download-models` to pre-download."
- A CLI command (`emailibrium --download-models`) allows users to pre-download all configured models while connected, for later offline use.

### Disk Space Management

- The `~/.emailibrium/models/` directory will consume approximately 100-500 MB depending on the number of models downloaded.
- A `make clean-models` target (and equivalent CLI command `emailibrium --clean-models`) removes all cached model files.
- The model manager does not automatically delete old models when switching. Users who want to reclaim space must run the cleanup command explicitly. This is intentional: keeping old models allows quick switching back without re-downloading.

### Configuration

```yaml
ai:
  embedding:
    onnx:
      # Model identifier (maps to fastembed EmbeddingModel enum)
      model: 'all-MiniLM-L6-v2'

      # Local cache directory for downloaded model files
      model_path: '~/.emailibrium/models'

      # Use INT8 quantized model variant (smaller download, slightly less accurate)
      quantized: false

      # Enable GPU acceleration (CoreML on macOS, CUDA on Linux/Windows)
      use_gpu: false

      # Pre-download model at application startup vs. lazy download on first embed() call
      download_on_startup: true

      # Show progress bar during model download
      show_download_progress: true

      # Number of threads for ONNX Runtime intra-op parallelism (0 = auto-detect)
      num_threads: 0
```

## Consequences

### Positive

- First-run experience is transparent: users see a progress bar and understand what is being downloaded
- SHA-256 verification protects against corrupted or tampered model files
- Model switching is safe: the system detects incompatible vectors and re-embeds automatically
- Offline users can pre-download models and then operate without any network access
- Multiple models can coexist on disk, enabling quick switching and A/B evaluation
- Re-embedding progress is exposed via API, allowing the frontend to display status

### Negative

- First run has a one-time download delay of approximately 10-30 seconds on broadband (longer on slow connections)
- Disk space used for model cache: 90-500 MB depending on models downloaded
- Model switching triggers a potentially expensive re-embedding operation (minutes to hours depending on email corpus size)
- During re-embedding, search quality is degraded (mixed old/new vectors in the HNSW index)
- The compile-time checksum manifest must be updated whenever new model versions are supported

### Neutral

- The `fastembed` crate handles most of the download and caching logic natively; the model manager primarily adds verification, model-switch detection, and re-embedding orchestration
- Quantized (INT8) models are available as a configuration option for users who prefer smaller downloads (~23 MB vs ~90 MB) at a minor quality cost
- The model storage layout follows the standard pattern used by Hugging Face Hub caches

## Alternatives Considered

### Require Manual Model Download

- **Pros**: No auto-download complexity, no first-run delay, user has full control
- **Cons**: Adds installation friction (contradicts zero-config design principle), users must know which model to download, where to place it, and how to verify integrity
- **Verdict**: Rejected. Auto-download with verification is the better trade-off for zero-config.

### Embed Model in Binary

- **Pros**: Single self-contained executable, no first-run download, no network dependency ever
- **Cons**: Increases binary size by 90 MB (FP32) or 23 MB (INT8), makes binary too large for package managers and CI/CD pipelines, model updates require binary rebuild and redistribution
- **Verdict**: Rejected. The binary size cost is prohibitive for distribution. Auto-download on first run is the standard approach used by `fastembed`, Ollama, and similar tools.

### No Model Switching Support

- **Pros**: Simpler implementation, no re-embedding logic, no stale vector tracking
- **Cons**: Users are locked into the initial model choice forever, cannot upgrade to better models as they become available, cannot switch to multilingual models when needed
- **Verdict**: Rejected. Model switching is essential for a pluggable architecture (ADR-002) to deliver on its promise.

### Automatic Old Model Cleanup

- **Pros**: Saves disk space automatically, no manual cleanup needed
- **Cons**: Deleting old models prevents quick switch-back, may surprise users who expect cached files to persist, risk of deleting a model that is still in use by a concurrent process
- **Verdict**: Rejected. Manual cleanup via `emailibrium --clean-models` is safer and more predictable.
