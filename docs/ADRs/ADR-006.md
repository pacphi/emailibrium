# ADR-006: Multi-Asset Content Extraction Pipeline

- **Status**: Proposed
- **Date**: 2026-03-23
- **Context**: The plan proposes extracting text from HTML bodies, inline images (OCR), attachments (PDF/DOCX/XLSX), and URLs. The research paper identifies reliability as a concern: OCR quality varies dramatically, PDF extraction fails on scanned docs, extraction quality is non-uniform across input types.
- **Decision**: Implement a 2-mode extraction pipeline (fast/deep) with quality scoring and graceful degradation per asset type.
- **Consequences**: Complex pipeline with many failure modes. Quality scoring adds overhead but prevents garbage-in-garbage-out. Graceful degradation ensures the pipeline never blocks on a single bad asset.
- **Alternatives Considered**: Text-only extraction (simpler, misses rich content), cloud extraction APIs (better quality, privacy violation), Apache Tika (Java, FFI overhead).
- **Research References**: OCR confidence scoring for noisy inputs; PDF text-vs-scanned detection heuristics; multi-collection vector search weighting strategies.

## Detailed Design

### Extraction Modes

| Mode | Trigger | Scope | Latency |
|------|---------|-------|---------|
| Fast | During sync | Text embedding only | ~5ms/email |
| Deep | Background Apalis job | Full multi-asset extraction | Runs after fast mode completes |

### Per-Asset Quality Scoring

- **OCR (ocrs crate)**: Confidence score threshold of 0.6. Below threshold, flag as low-confidence; still store but weight lower in search.
- **PDF**: Detect scanned vs text-based PDFs. Scanned PDFs route to OCR pipeline; text-based use pdf-extract directly.
- **HTML**: ammonia sanitization, then scraper structured extraction, then html2text clean output. Quality = ratio of extracted text to raw HTML length.
- **URLs**: Timeout at 5s per URL resolution, max 20 URLs per email, classify via domain matching + heuristics.

### Graceful Degradation

If any extraction step fails, log the error, continue the pipeline, and mark the asset as "extraction_failed" in metadata. No single asset failure should halt processing of an email.

### Asset-Specific Vector Collections

| Collection | Dimensions | Source |
|------------|-----------|--------|
| email_text | 384D | Always populated from body text |
| image_text | 384D | OCR output from inline images |
| image_visual | 512D | CLIP embedding from images |
| attachment_text | 384D | Extracted document text from PDFs, DOCX, XLSX |

### Multi-Collection Search Weights

| Collection | Weight |
|------------|--------|
| email_text (body) | 1.0 |
| attachment_text | 0.8 |
| image_text | 0.7 |
| image_visual | 0.5 |

Queries hit all collections; results are merged and ranked by weighted similarity scores.

### Performance Budget

| Email Type | Expected Latency |
|------------|-----------------|
| Text-only | ~5ms |
| Typical (2 images, 5 URLs) | ~520ms |
| Heavy (3 PDFs, 5 images) | ~2s |

## Options Considered

### Option 1: Text-Only Extraction
- **Pros**: Simple implementation, fast, no external dependencies for OCR/PDF parsing.
- **Cons**: Misses rich content in images, attachments, and linked resources. Significantly reduces search recall for users with attachment-heavy workflows.

### Option 2: Cloud Extraction APIs (e.g., Google Document AI, AWS Textract)
- **Pros**: Higher extraction quality, handles edge cases well, maintained by large teams.
- **Cons**: Violates local-first privacy model. Sends email content to third-party servers. Adds network dependency and per-call cost.

### Option 3: Apache Tika (via FFI)
- **Pros**: Battle-tested extraction for many file types, single unified API.
- **Cons**: Java dependency adds ~200MB to application size. FFI bridge introduces complexity and potential memory safety issues. JVM startup overhead.

### Option 4: 2-Mode Pipeline with Quality Scoring (Selected)
- **Pros**: Balances speed and depth. Quality scoring prevents low-quality extractions from polluting search. Graceful degradation ensures robustness. All processing stays local.
- **Cons**: Complex pipeline with many failure modes. Multiple vector collections increase storage. Quality thresholds require tuning.
