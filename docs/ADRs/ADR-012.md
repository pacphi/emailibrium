# ADR-012: Tiered AI Provider Architecture

- **Status**: Proposed
- **Date**: 2026-03-23
- **Extends**: ADR-002 (Embedding Model Selection Strategy), ADR-011 (ONNX Runtime as Default Embedding Provider)
- **Research References**: docs/research/llm-options.md -- Sections 6, 7, 8, 9

## Context

Emailibrium needs AI capabilities across three domains: embedding (vector search), classification (email triage), and chat (conversational AI for rule building and email summarization). Different users have fundamentally different privacy and quality requirements:

- A privacy-conscious user wants zero network calls and no external services.
- A power user wants local generative AI (Ollama) for chat and classification fallback while keeping all data on-machine.
- An enterprise user may accept cloud providers for maximum quality, with appropriate consent and audit controls.

The current architecture has no structured way to express these trade-offs. The embedding provider is configured independently, but there is no generative provider abstraction, no consent flow for cloud usage, and no audit logging for data that leaves the machine.

The research in docs/research/llm-options.md proposes a 3-tier architecture that gives users full control over their privacy/quality trade-off, with embedding and generative providers configured independently.

## Decision

Implement a 3-tier AI provider system where embedding and generative capabilities are configured independently.

### Tier 0 -- Zero Config (Default)

**Target user:** Anyone who installs Emailibrium. No AI knowledge required.

| Component               | Implementation                 | Details                                                   |
| ----------------------- | ------------------------------ | --------------------------------------------------------- |
| Text embedding          | `fastembed` via ONNX (ADR-011) | `all-MiniLM-L6-v2`, 384D, auto-downloads                  |
| Classification          | Vector centroid matching only  | Cosine similarity to learned category centroids           |
| Low-confidence fallback | Rule-based heuristics          | Sender domain patterns, keyword matching, header analysis |
| Chat                    | Disabled                       | Template-based suggestions only; no generative model      |
| Image embedding         | Deferred                       | Not available at Tier 0                                   |
| Network calls           | Zero                           | All inference runs in-process                             |
| External dependencies   | Zero                           | No Ollama, no cloud APIs, no Docker                       |

**Rationale for no generative model at Tier 0:** The research finds that 85-95% of emails can be classified by cosine similarity to category centroids alone. The remaining 5-15% of ambiguous emails (confidence 0.5-0.7) can be handled by rule-based heuristics (sender domain, keyword matching, header analysis) rather than requiring a generative model. This keeps the default truly zero-dependency.

### Tier 1 -- Local Enhanced (Opt-in)

**Target user:** Power users who want generative AI features while keeping all data local.

| Component               | Implementation                                                            | Details                                                    |
| ----------------------- | ------------------------------------------------------------------------- | ---------------------------------------------------------- |
| Text embedding          | `fastembed` via ONNX                                                      | Same as Tier 0 (ONNX is faster than Ollama for embeddings) |
| Classification fallback | Ollama generative model                                                   | Called only when centroid confidence < 0.7                 |
| Chat                    | Ollama with user's choice of model                                        | Default: `llama3.2:3b` or `phi-3.5-mini` (2.5-3 GB RAM)    |
| Image embedding         | CLIP ViT-B-32 via ONNX                                                    | 512D, separate HNSW collection                             |
| Network calls           | Zero (all localhost)                                                      | Ollama runs as a local process                             |
| External dependencies   | Ollama installed and running                                              | User must `ollama pull` the desired model                  |
| User action             | Install Ollama, pull model, set `generative.provider: "ollama"` in config |

### Tier 2 -- Cloud (Explicit Consent Required)

**Target user:** Users who want maximum quality and accept the privacy trade-off.

| Component             | Implementation                                                     | Details                                                   |
| --------------------- | ------------------------------------------------------------------ | --------------------------------------------------------- |
| Text embedding        | Optionally upgrade to cloud API                                    | OpenAI `text-embedding-3-small` or Cohere `embed-v4`      |
| Classification        | Cloud LLM                                                          | Claude Haiku or GPT-4o-mini for ambiguous emails          |
| Chat                  | Cloud LLM                                                          | Claude Sonnet or GPT-4o for full conversational AI        |
| Network calls         | Per-inference API calls                                            | Email text sent to cloud provider                         |
| External dependencies | API key required                                                   | Set via environment variable, never stored in config file |
| User action           | Set API key env var, set provider in config, accept consent dialog |

**Mandatory safeguards for Tier 2:**

1. **Consent dialog**: Before the first cloud API call, display a clear message: "Your email content will be sent to [provider] for processing. This data will leave your machine." User must explicitly acknowledge.
2. **Audit logging**: Every cloud API call is logged to an audit table with timestamp, provider, model, token count, and a truncated hash of the input (not the full content).
3. **Content minimization**: Send only the minimum text needed (subject + sender + truncated body), never full email bodies or attachment contents.
4. **API key hygiene**: API keys are read only from environment variables, never stored in YAML config or SQLite database.

### Independent Configuration

Embedding and generative providers are configured independently. This allows the common and recommended pattern: local ONNX for embeddings (fast, private) combined with cloud for chat (better quality for conversational AI).

```yaml
ai:
  embedding:
    provider: 'onnx' # independent -- controls vector generation
  generative:
    provider: 'none' # independent -- controls classification fallback and chat
```

Valid combinations include:

| Embedding | Generative | Use Case                                                              |
| --------- | ---------- | --------------------------------------------------------------------- |
| `onnx`    | `none`     | Default: full privacy, embedding-only AI                              |
| `onnx`    | `ollama`   | Local power user: ONNX embeddings + Ollama chat                       |
| `onnx`    | `cloud`    | Privacy-conscious with cloud chat: local embeddings, cloud generative |
| `ollama`  | `ollama`   | Full Ollama: all AI through Ollama (not recommended for embeddings)   |
| `cloud`   | `cloud`    | Full cloud: maximum quality, minimum privacy                          |

### Provider Resolution Logic

```
1. Read ai.embedding.provider (default: "onnx")
2. If "onnx": initialize fastembed, auto-download model if needed
3. If "ollama": check Ollama reachability; if unreachable, fall back to ONNX with warning
4. If "cloud": check consent + API key; if missing, fall back to ONNX with error

5. Read ai.generative.provider (default: "none")
6. If "none": classification uses centroid-only + rule-based fallback; chat disabled
7. If "ollama": check Ollama reachability; if unreachable, fall back to "none" with warning
8. If "cloud": check consent + API key; if missing, fall back to "none" with error
```

Fallback is always toward greater privacy (cloud -> ollama -> onnx -> mock for embedding; cloud -> ollama -> none for generative).

## Consequences

### Positive

- Users get full control over their privacy/quality trade-off at a granular level
- Default installation requires zero configuration and zero external dependencies
- Independent embedding/generative configuration enables the optimal pattern (local embeddings + cloud chat)
- Consent and audit mechanisms satisfy GDPR requirements for Tier 2
- Graceful degradation: if a configured provider is unavailable, the system falls back to a more private option rather than failing

### Negative

- Configuration complexity increases: users must understand the distinction between embedding and generative providers
- Must implement consent flow in the frontend (dialog, persistence of consent state)
- Audit logging for Tier 2 adds storage overhead (one row per cloud API call)
- Three tiers means three code paths to test and maintain for each AI feature
- Documentation burden: must clearly explain each tier's capabilities and trade-offs

### Neutral

- Tier 0 and Tier 1 have identical embedding behavior (both use ONNX via fastembed)
- The tiered architecture does not affect non-AI features (IMAP sync, FTS5, UI)
- Users can change tiers at any time via configuration; no data migration required (except re-embedding if switching embedding models with different dimensions)

## Alternatives Considered

### Single Provider for Everything

- **Pros**: Simpler configuration, one code path, easier to test
- **Cons**: Forces users into a single privacy/quality trade-off; cannot use local embeddings with cloud chat; cannot offer zero-config default alongside cloud capabilities
- **Verdict**: Rejected. The privacy-first design principle requires fine-grained control.

### Auto-Detect Provider (Magic)

- **Pros**: Zero configuration for all tiers (detect Ollama presence, detect API keys)
- **Cons**: Magic is fragile (Ollama installed but not running? API key set but consent not granted?), hard to debug, violates principle of least surprise, could accidentally send data to cloud if an API key env var is set for another application
- **Verdict**: Rejected. Explicit configuration is safer and more predictable.

### Cloud-First with Local Fallback

- **Pros**: Best quality by default, local as a fallback for offline use
- **Cons**: Violates the privacy-first design principle, sends email data to cloud by default, requires API key for basic functionality
- **Verdict**: Rejected. Fundamentally incompatible with Emailibrium's privacy-first positioning.

### Two Tiers Only (Local vs. Cloud)

- **Pros**: Simpler mental model, fewer configuration options
- **Cons**: Collapses the distinction between "zero-config ONNX" and "Ollama with generative models"; users who want ONNX embeddings without Ollama have no clear path; users who want Ollama chat without cloud have no distinct tier
- **Verdict**: Rejected. Three tiers map naturally to the three levels of external dependency (none, local service, cloud API).
