# ADR-005: Tauri 2.0 to Web SPA Migration

- **Status**: Proposed
- **Date**: 2026-03-23

## Context

The original Emailibrium plan specified Tauri 2.0 as the desktop application framework. This decision is being revisited in favor of a pure web SPA (Single Page Application) built with React and TypeScript. The migration is motivated by distribution friction (desktop apps require download, install, and platform-specific builds), cross-platform limitations (Tauri requires separate testing on macOS, Windows, Linux), and development velocity (web HMR cycles are approximately 10x faster than Tauri rebuild cycles).

The Emailibrium team is primarily TypeScript-skilled on the frontend, and the Rust backend can be exposed via HTTP API regardless of the frontend technology. A web SPA communicating with a local Rust backend server provides the same local-first architecture without the desktop app wrapper.

## Decision

Build a pure web SPA with React and TypeScript. No Electron. No Tauri. Optionally wrap as a PWA (Progressive Web App) for offline capability and native-like experience. The Rust backend runs as a local HTTP server that the SPA communicates with.

### Architecture

```
Browser (React SPA)
    |
    | HTTP/WebSocket (localhost)
    |
Rust Backend Server (actix-web or axum)
    |
    +-- SQLite (FTS5)
    +-- RuVector (HNSW)
    +-- Email sync engine
```

### Security Implications and Mitigations

1. **Credential storage**: Browser secure storage is weaker than native OS keychain.
   - Mitigation: Use Web Crypto API with AES-GCM encryption and non-extractable CryptoKey objects
   - OAuth tokens stored in encrypted IndexedDB, never in localStorage (vulnerable to XSS)
   - Refresh tokens held only in the Rust backend's memory or encrypted on-disk store

2. **XSS attack surface**: Web applications are inherently exposed to cross-site scripting.
   - Mitigation: Strict Content Security Policy (CSP) headers
   - All user-generated content (email bodies, subjects) sanitized with DOMPurify before rendering
   - No use of `dangerouslySetInnerHTML` without sanitization
   - CSP: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self' localhost:*`

3. **CORS configuration**: The local Rust backend must serve appropriate CORS headers.
   - Mitigation: No wildcard origins -- explicitly allow only `http://localhost:*` and the specific serving origin
   - Credentials mode: `include` only for authenticated endpoints

4. **Local network exposure**: A local HTTP server could be accessed by other applications or pages.
   - Mitigation: Bearer token authentication on all API endpoints, token generated at startup and passed to the SPA
   - Bind to `127.0.0.1` only (not `0.0.0.0`)

### Offline Capability

- PWA Service Worker caches the application shell and static assets
- IndexedDB stores recent email data for offline search
- Background sync queues operations (mark read, categorize, archive) performed offline
- Sync reconciliation when connectivity to the backend is restored

### Performance Budget

| Metric                         | Target             | Enforcement             |
| ------------------------------ | ------------------ | ----------------------- |
| Largest Contentful Paint (LCP) | < 1.5s             | Lighthouse CI gate      |
| First Input Delay (FID)        | < 100ms            | Lighthouse CI gate      |
| Cumulative Layout Shift (CLS)  | < 0.1              | Lighthouse CI gate      |

| Route-based code splitting     | All feature routes | Build-time verification |

### Accessibility

- WCAG 2.1 AA compliance required
- axe-core integrated into CI pipeline for automated accessibility testing
- Keyboard navigation for all interactive elements
- Screen reader compatibility tested with VoiceOver (macOS) and NVDA (Windows)
- Color contrast ratio minimum 4.5:1 for normal text, 3:1 for large text

### Technology Stack

- **Framework**: React 18+ with TypeScript
- **State management**: Zustand (lightweight, TypeScript-native)
- **Routing**: React Router v6 with lazy-loaded routes
- **Styling**: Tailwind CSS (utility-first, tree-shakeable)
- **Build**: Vite (fast HMR, optimized production builds)
- **Testing**: Vitest + React Testing Library + Playwright (E2E)

## Consequences

### Positive

- Zero-install access via URL -- users open a browser and start using the application
- Any browser, any platform (macOS, Windows, Linux, ChromeOS, tablets, phones)
- 10x faster development iteration with Vite HMR vs. Tauri rebuild cycles
- Simplified CI/CD -- no platform-specific builds, signing, or notarization
- Pure TypeScript frontend team can work independently from Rust backend team
- PWA wrapping provides offline support and "Add to Home Screen" for native-like experience
- Single codebase serves desktop and mobile form factors

### Negative

- Lose native OS integration: no system tray icon, no global keyboard shortcuts, no direct file system access, no OS notification center integration
- Browser secure storage is fundamentally less secure than OS keychain -- a compromised browser extension could access stored credentials
- Users must keep a browser open (or use PWA) rather than having a dedicated desktop window
- Local HTTP server must be started separately (or auto-started via OS service) before the SPA is usable
- Some users may perceive a web app as less "serious" or "professional" than a native desktop application

### Neutral

- The Rust backend API design is the same regardless of whether Tauri or a web SPA consumes it -- this decision does not affect backend architecture
- PWA install prompts vary by browser and OS, providing an inconsistent "install" experience
- The local-first privacy model is preserved -- all data stays on the user's machine, the SPA communicates only with localhost

## Alternatives Considered

### Keep Tauri 2.0

- **Pros**: Native OS integration (system tray, global shortcuts, file system, keychain), smaller bundle than Electron, Rust backend is naturally integrated, native window management
- **Cons**: Requires platform-specific builds and signing (Apple notarization, Windows code signing), installation friction (download, trust, install), WebView rendering inconsistencies across platforms, slower development cycle (Rust rebuild required for backend changes)
- **Verdict**: Better native integration but worse distribution and development velocity. The distribution friction is a significant barrier to adoption for a new product.

### Electron

- **Pros**: Mature ecosystem, extensive native API surface, Chromium guarantees consistent rendering, large community and plugin ecosystem
- **Cons**: 150-300MB base binary size, high memory usage (each instance runs a full Chromium), performance overhead, security concerns with Node.js integration in renderer, still requires platform-specific distribution
- **Verdict**: The weight and resource consumption of Electron are disproportionate for an application that can work as a web SPA.

### Capacitor (Ionic)

- **Pros**: Web-first with native plugins, good for mobile deployment, single codebase for web and mobile
- **Cons**: Mobile-focused tooling adds unnecessary complexity for a desktop-first product, smaller plugin ecosystem than Electron, additional abstraction layer, community is smaller
- **Verdict**: Adds complexity for mobile support that is not a current priority. If mobile becomes important, PWA already provides basic mobile support.

### Flutter Web

- **Pros**: Single codebase for web, mobile, and desktop, high-performance rendering via Skia/Impeller
- **Cons**: Dart language (team expertise is TypeScript), large WASM payload for web, limited web SEO/accessibility tooling, different development paradigm from the existing frontend skills
- **Verdict**: Language mismatch with team skills makes this impractical without significant ramp-up time.
