# ADR-008: Privacy Architecture and Embedding Security

- **Status**: Proposed
- **Date**: 2026-03-23
- **Context**: The research paper identifies embedding invertibility as an unaddressed risk (Morris et al. 2023 showed text can be partially recovered from sentence embeddings). The plan covers OWASP secrets management and Web Crypto but not vector store security.
- **Decision**: Treat vector embeddings as sensitive data equivalent to email content. Implement encryption at rest, access controls, and embedding invertibility mitigations.
- **Consequences**: Encryption at rest adds ~5-10% overhead on vector operations. Noise injection slightly degrades search quality. Key management adds complexity to startup flow. But these are necessary for handling PII-adjacent data.
- **Alternatives Considered**: No encryption (faster, but negligent for email data), full homomorphic encryption (theoretically ideal, computationally infeasible), server-side storage (shifts trust model).
- **Research References**: Morris et al. 2023 (text recovery from sentence embeddings); OWASP secrets management guidelines; differential privacy techniques for vector stores.

## Detailed Design

### Vector Store Encryption at Rest

1. REDB storage encrypted at rest using AES-256-GCM.
2. Encryption key derived from user's master password via Argon2id KDF.
3. Key never stored on disk -- derived at application startup, held in memory only.
4. On application shutdown, key is zeroed from memory using the `zeroize` crate.

### Browser-Side Security

1. OAuth tokens stored via Web Crypto API AES-GCM with a non-extractable CryptoKey in IndexedDB.
2. Never use localStorage for tokens (XSS-accessible).
3. Content Security Policy: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'`.

### Embedding Invertibility Mitigations

1. Add calibrated Gaussian noise (sigma=0.01) to stored embeddings (differential privacy lite).
2. Validate that the noise level does not degrade recall@10 by more than 1% (empirical testing required).
3. Document in privacy policy: "Embeddings are mathematical representations that cannot be directly read as email text, but may contain information derived from email content."

### Metadata Protection

1. SQLite database encrypted via SQLCipher (or sqlx with encryption extension).
2. Full disk encryption recommended in documentation.

### Audit Logging

1. Log all vector store access (who, what query, when) to an append-only audit table.
2. Retention: 90 days default, configurable.

### Device Loss Mitigation

1. Remote wipe capability via account settings (deletes local vector store + SQLite).
2. Requires an authenticated session to trigger.

## Options Considered

### Option 1: No Encryption
- **Pros**: Fastest possible vector operations. Simplest implementation. No key management complexity.
- **Cons**: Negligent for an application handling email data. A stolen device exposes all email content and embeddings. Unacceptable risk profile.

### Option 2: Full Homomorphic Encryption (FHE)
- **Pros**: Theoretically ideal -- compute on encrypted data without decryption. Would allow search over encrypted embeddings.
- **Cons**: Computationally infeasible for real-time search. Current FHE schemes add 1000x+ overhead. Not practical for a desktop application.

### Option 3: Server-Side Storage with E2E Encryption
- **Pros**: Enables cross-device sync. Server never sees plaintext.
- **Cons**: Shifts trust model away from local-first. Adds network dependency. Server infrastructure costs. Contradicts the core architecture principle of local processing.

### Option 4: Encryption at Rest + Noise Injection (Selected)
- **Pros**: Practical balance of security and performance. AES-256-GCM is hardware-accelerated on modern CPUs. Noise injection adds a layer of defense against embedding inversion. Key derivation from master password avoids key storage.
- **Cons**: ~5-10% overhead on vector operations. Noise injection slightly reduces search precision. Master password required at every application startup. If master password is lost, data is unrecoverable.
