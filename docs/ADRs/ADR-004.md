# ADR-004: SONA Adaptive Learning Model Specification

- **Status**: Proposed
- **Date**: 2026-03-23

## Context

Emailibrium's search quality should improve over time based on user behavior. The plan describes a 3-tier adaptive learning system called SONA (Self-Organizing Neural Adaptation) that adjusts search re-ranking based on user feedback signals. However, the research paper (docs/research/initial.md) identifies SONA as the biggest specification gap in the plan: the conceptual description lacks formal update rules, convergence guarantees, learning rates, bounds on centroid drift, and safeguards against feedback loops or catastrophic forgetting.

Without formal specification, SONA risks: (1) unbounded centroid drift from noisy feedback, (2) catastrophic forgetting where new patterns overwrite valuable old patterns, (3) position bias amplification where top-ranked results accumulate clicks regardless of relevance, and (4) degenerate collapse where all categories converge to similar centroids.

## Decision

Formalize SONA with explicit mathematical update rules, bounded parameters, and degenerate-case safeguards across all three tiers.

### Tier 1: Instant Adaptation (per-interaction, <1ms)

Triggered when a user provides explicit feedback (reclassifies an email, marks as relevant/irrelevant).

**Centroid update rule (positive feedback):**

```
mu_b <- (1 - alpha) * mu_b + alpha * v(e)
```

Where:

- `mu_b` = centroid of target category `b`
- `v(e)` = embedding vector of email `e`
- `alpha = 0.05` (positive learning rate)

**Centroid update rule (negative feedback -- email removed from category `a`):**

```
mu_a <- (1 - beta) * mu_a + beta * (-0.3 * v(e))
```

Where:

- `beta = 0.02` (negative learning rate, lower to prevent overcorrection)
- The -0.3 scaling factor provides a mild repulsive effect

**Bound:** `||delta_mu|| <= max_shift` where `max_shift = 0.1 * ||mu||`. A single feedback event cannot move a centroid by more than 10% of its magnitude.

### Tier 2: Session Adaptation (within-session, accumulated)

Accumulates implicit signals (clicks, time-spent, scroll depth) within a single search session.

**Session preference vector:**

```
p_s = mean(v(clicked_emails)) - mean(v(skipped_emails))
```

**Re-ranking boost:**

```
score'(d) = score(d) + gamma * cos(v(d), p_s)
```

Where:

- `gamma = 0.15` (session boost weight)
- `cos()` = cosine similarity

**Lifecycle:** Session preference vector `p_s` is computed incrementally and reset when the session ends. It is not persisted.

### Tier 3: Long-term Consolidation (background, periodic)

**Hourly consolidation:**

- Run mini-batch K-means on recently unassigned/low-confidence emails
- Reclassify emails with confidence < 0.6 based on updated centroids
- Lightweight: processes only new/changed emails since last run

**Daily consolidation:**

- Full HDBSCAN re-clustering of the entire email corpus
- Recompute all centroids from complete feedback history
- Apply SONA BaseLoRA adaptation with EWC++ (Elastic Weight Consolidation) to prevent catastrophic forgetting

**EWC++ formulation:**

```
L_total = L_task + lambda * SUM_i( F_i * (theta_i - theta_i*)^2 )
```

Where:

- `L_task` = current task loss
- `F_i` = Fisher information matrix diagonal for parameter `i` (importance weight)
- `theta_i*` = optimal parameters from previous consolidation
- `lambda = 0.4` (regularization strength, tunable)

### Safeguards

1. **Cold start protection**: Centroid updates only take effect after a minimum of 10 feedback events per category. Before this threshold, use static centroids from initial clustering.

2. **Centroid drift alarm**: If any centroid moves more than 20% from its initial position (measured by cosine distance), log a warning and flag for review. At 40% drift, freeze updates for that category pending manual review.

3. **A/B evaluation**: Serve 10% of queries without SONA re-ranking (control group). Compare NDCG@10 between SONA and non-SONA groups weekly. If SONA group underperforms for 2 consecutive weeks, trigger automatic rollback.

4. **Centroid snapshots**: Persist complete centroid state daily. Enable 1-click rollback to any snapshot from the past 30 days.

5. **Position bias mitigation**: If a user clicks the rank-1 result more than 95% of the time over a 50-query window, discount those click signals by 0.5x. Position bias is a known confounder in implicit feedback (Joachims et al. 2007).

6. **Degenerate collapse detection**: Monitor pairwise cosine distances between all category centroids. If minimum pairwise distance drops below 0.15, alert that categories are merging and halt Tier 1 updates.

## Consequences

### Positive

- Formal mathematical specification enables unit testing of update rules with known inputs/outputs
- Bounded updates prevent single feedback events from corrupting centroids
- EWC++ protects against catastrophic forgetting during daily consolidation
- A/B evaluation provides ongoing empirical validation that SONA improves quality
- Centroid snapshots with rollback provide a safety net for unexpected behavior
- Position bias mitigation addresses a well-known source of noise in click feedback

### Negative

- EWC++ adds computational complexity to daily consolidation (Fisher information matrix computation)
- A/B evaluation infrastructure requires traffic splitting and metric tracking
- 6 separate safeguard mechanisms increase implementation and testing surface area
- The 10-event cold start threshold delays personalization for new users or new categories
- Hyperparameters (alpha, beta, gamma, lambda, max_shift, thresholds) require tuning on real data

### Neutral

- Tier 1 and Tier 2 are lightweight and add negligible latency to the search path
- Tier 3 runs in background and does not affect interactive performance
- Safeguard thresholds (20% drift, 95% click rate, 0.15 cosine distance) are initial values subject to calibration

## Alternatives Considered

### No Learning (Static Search)

- **Pros**: Simplest implementation, deterministic behavior, no risk of degradation over time
- **Cons**: Search quality never improves, does not adapt to individual user patterns, fails to capture evolving email usage patterns
- **Verdict**: Unacceptable for a product whose value proposition includes personalized email management

### Full Model Fine-tuning

- **Pros**: Potentially highest quality adaptation, end-to-end optimization
- **Cons**: Computationally expensive for local execution (requires GPU or hours of CPU time), risk of catastrophic forgetting without careful regularization, requires significant labeled data
- **Verdict**: Too resource-intensive for a local-first application on consumer hardware

### Collaborative Filtering

- **Pros**: Leverages patterns across users, well-understood recommendation technique
- **Cons**: Requires multi-user data that a local-first application does not have, privacy implications of sharing user behavior, cold-start problem at the population level
- **Verdict**: Fundamentally incompatible with the local-first, privacy-preserving architecture

### Simple Click-Through Rate Boosting

- **Pros**: Very simple to implement, well-understood, no centroid mathematics
- **Cons**: Highly susceptible to position bias, no semantic understanding of why items are clicked, requires large click volumes to be statistically meaningful
- **Verdict**: Too crude for the nuanced re-ranking needed in email search
