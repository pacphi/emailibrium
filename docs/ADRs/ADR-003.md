# ADR-003: RuVector as Primary Vector Database

- **Status**: Proposed
- **Date**: 2026-03-23

## Context

Emailibrium requires a vector database for storing and searching email embeddings via HNSW (Hierarchical Navigable Small World) indices. The system runs locally on user machines, so the vector DB must be embeddable (no separate server process), performant on consumer hardware, and reliable under unexpected shutdowns (laptop lid close, power loss).

RuVector is a Rust-native vector database offering HNSW search with SIMD acceleration, SONA adaptive re-ranking, and GNN-based features. It uses REDB as its storage backend. The primary concern, flagged in the research paper (docs/research/initial.md), is maturity: RuVector lacks a long-term production track record, REDB durability under crash scenarios is unproven at scale, and concurrent read-write correctness has not been independently validated.

## Decision

Use RuVector as the primary vector database, protected by a `VectorStore` facade trait that enables swapping to an alternative (Qdrant embedded, LanceDB) if RuVector proves insufficient. Implement defensive measures including backup to SQLite, write-ahead checkpointing, and a dedicated crash-recovery test suite.

### Architecture

```
SearchService --> VectorStore (trait)
                      |
                      +-- RuVectorStore (default implementation)
                      +-- QdrantStore (fallback, behind feature flag)
                      +-- SqliteVectorStore (emergency fallback, slow but reliable)
```

### VectorStore Facade

```rust
#[async_trait]
trait VectorStore {
    async fn insert(&self, id: EmailId, vector: &[f32]) -> Result<()>;
    async fn insert_batch(&self, items: &[(EmailId, Vec<f32>)]) -> Result<()>;
    async fn search(&self, query: &[f32], top_k: usize) -> Result<Vec<(EmailId, f32)>>;
    async fn delete(&self, id: EmailId) -> Result<()>;
    async fn count(&self) -> Result<usize>;
    async fn health_check(&self) -> Result<HealthStatus>;
}
```

### RuVector Advantages

- Zero FFI overhead (pure Rust, same language as Emailibrium core)
- Built-in HNSW with SIMD acceleration for search performance
- SONA adaptive re-ranking integrated at the vector DB level
- GNN-based features for graph-aware email relationship modeling
- REDB provides ACID transactions with write-ahead logging

### Risk Mitigations

1. **VectorStore facade trait**: Enables swap to Qdrant or other backend without cascading changes to consumers. Indirection cost is ~1 microsecond per call (negligible vs. search latency).

2. **Periodic backup to SQLite**: Every 1000 write operations, persist a snapshot of (email_id, vector_blob) pairs to a SQLite table. This enables full index rebuild from scratch if REDB corruption occurs.

3. **Write-ahead checkpoint**: Force REDB checkpoint every 1000 operations to bound recovery time and data loss window.

4. **Crash recovery test suite**: Automated tests that simulate crashes during writes:
   - `kill -9` during batch insert, verify index integrity on restart
   - `kill -9` during delete, verify no phantom entries
   - Concurrent read during write crash, verify reader does not see partial state
   - Power-loss simulation (truncate WAL file mid-write), verify recovery

5. **Comparison benchmark suite**: Before Phase 2.5, run head-to-head comparison of RuVector vs. Qdrant (embedded mode) measuring:
   - Write throughput (vectors/second)
   - Search latency (p50, p95, p99)
   - Recall@10 at various index sizes (1K, 10K, 100K, 500K vectors)
   - Crash recovery time
   - Memory usage under load

### Scale Targets

- Must handle 100K+ vectors under concurrent read/write load
- Search latency target: <2ms for top-100 candidates at 100K index size
- Write throughput target: >1000 vectors/second for batch indexing

## Consequences

### Positive

- Pure Rust integration eliminates FFI complexity, build system issues, and cross-platform linking problems
- SONA and GNN features are available natively without additional integration work
- Facade pattern provides a clean escape hatch if RuVector proves unreliable
- SQLite backup enables disaster recovery without data loss
- Crash recovery tests provide confidence before production deployment

### Negative

- RuVector is less battle-tested than Qdrant or Milvus -- unknown failure modes may emerge under real-world usage patterns
- REDB WAL behavior under extreme conditions (disk full, filesystem corruption) is not well-documented
- Backup to SQLite adds storage overhead (~150MB for 100K 384D vectors)
- Maintaining the facade trait and multiple implementations increases code surface area
- Benchmark suite development is a significant upfront investment (estimated 2-3 days)

### Neutral

- If RuVector works reliably, the facade and backup infrastructure become low-cost insurance that is never exercised
- The comparison benchmark results will inform future architecture decisions regardless of outcome

## Alternatives Considered

### Qdrant (Embedded Mode)

- **Pros**: Production-proven at scale (used by major companies), Rust-native, mature HNSW implementation, well-documented crash recovery, active community and commercial backing
- **Cons**: Heavier dependency (~15MB binary addition), embedded mode is less mature than server mode, does not include SONA/GNN features natively, would require custom re-ranking layer
- **Verdict**: Primary fallback candidate. If RuVector fails benchmarks or crash tests, migrate to Qdrant via the VectorStore facade.

### Milvus

- **Pros**: Feature-rich, supports multiple index types (IVF, HNSW, DiskANN), enterprise-grade, extensive filtering capabilities
- **Cons**: Written in Go/C++, requires FFI from Rust, designed as a server (not embeddable), massive dependency for a local-first application
- **Verdict**: Overengineered for the local-first use case. Not suitable.

### LanceDB

- **Pros**: Rust-native, columnar storage (efficient for filtered queries), disk-based (low memory), Apache Arrow integration
- **Cons**: Newer project with limited production track record (similar maturity concern as RuVector), different query model, less community adoption
- **Verdict**: Interesting but trades one maturity risk for another. Monitor for future evaluation.

### FAISS

- **Pros**: Meta-backed, extensively benchmarked, supports GPU acceleration, industry standard for vector search research
- **Cons**: C++ library requiring FFI bindings, complex build system (especially on macOS/Windows), no built-in persistence (must manage serialization), GPL-licensed components
- **Verdict**: FFI complexity and lack of built-in persistence make it unsuitable for an embedded local-first application.
